物体条件付き深度精緻化
実装計画

2段階学習で深度推定を行う研究の実装ロードマップです。 学習1回目でコア（全体）深度を学習し、学習2回目で YOLO 検出結果を条件に 物体領域の詳細深度を学習します。

学習1: コア予測モデル 学習2: 詳細予測モデル 方式A: オフライン固定データセット 推定所要: 約 4〜5 週間

1. 研究の目的

単眼深度推定モデル（Lotus-D）の出力を、物体単位でより詳細・正確な深度に改善する。

1

学習1: コア予測

RGB のみで全体の深度を予測するモデルを学習

2

YOLO で物体検出

推論時に RGB から bbox・クラス・マスクを取得（詳細予測の前提）

3

学習2: 詳細予測

コア深度 + YOLO 条件を入力し、物体領域を精緻化するモデルを学習

YOLO の入力について 物体検出は RGB 画像 に対して行います。深度カラーマップではなく RGB → YOLO → 詳細予測 の流れです。
旧計画との違い 以前の案は「公式 Lotus-D への学習1回（微調整のみ）」でした。 本計画は 学習を2回 に分け、2回目で YOLO 由来の物体条件を使います。

2. 2段階学習の設計（重要）

学習は 2回 行います。1回目と2回目は役割・入力・出力が異なる別モデルです。

学習1回目: コア予測 学習2回目: 詳細予測
役割 画像全体の深度を予測（ベースライン） 物体領域の深度を精緻化（ROI 重視）
入力 RGB のみ（UNet 4ch） RGB + pre-depth + 有効マスク + class map（UNet 9〜13ch）
学習データ RGB + GT 深度（Hypersim 等） 方式A: 事前構築した固定データセット（下記参照）
YOLO の使用 使わない 推論時: 毎回実行 / 学習時: 事前生成済みデータを読むだけ
初期重み SD2 Base または公式 Lotus-D 学習1のチェックポイント（conv_in ゼロ拡張）
学習スクリプト train_lotus_d_core.ps1（新規） train_lotus_d_detail.ps1（新規、旧 pre_depth を改名）
出力 output/core-model/ output/detail-model/

学習2のデータ準備 — 方式A（オフライン固定データセット）

学習2の前に、全学習データに対して推論パイプラインと同じ処理を1回だけ実行し、 固定データセットとして保存します。学習ループ内で YOLO やコアモデルを回しません。

方式A を採用する理由 推論時と同一の条件（YOLO 検出 → コアモデル深度 → 整合 → pre-depth）で学習できるため、 最も確実です。学習中の擬似合成（GT 深度の劣化）は使いません。

オフライン前処理の3ステップ

Step 1 YOLO 全データ実行

↓
Step 2 コアモデルで pre-depth 生成
global + crop 深度 → lstsq 整合 → pre_depth.npy
↓
Step 3 class_map 生成
YOLO クラス ID → class_map.npy
↓
固定学習データセット完成

データセットのディレクトリ構成

推論時 vs 学習時

推論時（本番）

  • RGB → YOLO（毎回実行）
  • コアモデル → グローバル + クロップ深度
  • lstsq 整合 → pre-depth + class map をその場で構築
  • 詳細モデル → 最終深度

学習時（方式A）

  • 事前保存済みの pre-depth / class_map / valid_mask を読む
  • GT 深度を教師ラベルとして学習
  • YOLO・コアモデルは学習ループ外（高速・再現性◎）
  • 条件ドロップアウト（確率30%）のみ学習時に適用

学習2の1ステップ（方式A）

# 学習ループ内（方式A） batch = dataloader.next() RGB ← 元画像 GT深度 ← 教師ラベル pre_depth ← 事前生成済み .npy を読む（推論パイプラインと同一） valid_mask ← 事前生成済み .npy を読む class_map ← 事前生成済み .npy を読む → 詳細モデル (UNet 9〜13ch) で GT深度を予測 → ROI 重み付き損失 + 勾配損失
なぜ2段階に分けるか コア予測と詳細予測を同時に学ぶと、物体条件への過依存や全体深度の劣化が起きやすい。 まず全体深度を安定させ、次に物体条件の融合だけを学ぶ方が安全です。 詳細モデルはゼロ初期化により、条件なしではコアモデルと同等の出力から始まります。

5. 2つのモデルのアーキテクチャ

モデルA: コア予測（学習1回目）

コアモデル UNet 入力 (4ch): RGB latent (4) 役割: 画像全体の深度。YOLO は使わない。 初期重み: stabilityai/stable-diffusion-2-base または jingheya/lotus-depth-d-v2-0-disparity 学習: 標準の train_lotus_d.py（--enable_pre_depth_fusion なし）

モデルB: 詳細予測（学習2回目）

コアモデルの重みから開始し、UNet conv_in をゼロ初期化で拡張する。

① pre-depth マップ

  • 整合済みローカル深度を貼り込んだマップ
  • VAE latent 4ch + 有効マスク 1ch
  • 既存実装あり

② class map（新規）

  • 各画素の物体クラス ID（0 = 背景）
  • 1ch クラスマップ → VAE encode → 4ch latent
  • 重なりは score の高い検出を優先

③ object stats（任意・将来）

  • 各 bbox の [class, cx, cy, w, h, depth_median]
  • 最大 K 物体を MLP でベクトル化
  • class_labels に加算する方式

詳細モデルへの条件入力（YOLO 由来）

現状 (9ch): RGB latent (4) + pre-depth latent (4) + valid mask (1) 追加後 (13ch): RGB latent (4) + pre-depth latent (4) + valid mask (1) + class map latent (4) 詳細モデル UNet 入力 (9ch 基本): RGB latent (4) + pre-depth latent (4) + valid mask (1) 拡張版 (13ch): 上記 + class map latent (4) ← YOLO のクラス ID から生成 ※ 追加分は ゼロ初期化 → 学習2開始時はコアモデルと同一出力 ※ pre-depth は YOLO bbox → コアモデル推定 → 整合合成 で構築（推論時）
ゼロ初期化の意味（学習2） 詳細モデルは学習開始時、物体条件なしではコアモデルと同じ深度を出力します。 学習を進めると YOLO 由来の pre-depth / class 情報を物体領域の精緻化に使い始めます。

6. フェーズ別実装計画

実装フェーズは2段階学習に対応して再構成しています。

0 ベースライン確認 1〜2 日

公式モデルと既存コードの動作確認。2段階学習の比較基準を固定。

  • 公式 lotus-depth-d-v2-0-disparity で NYUv2 評価
  • infer_roi_fusion.py で ROI 融合の上限確認
  • ゼロ初期化等価性テスト（check_pre_depth_zero_init.py）
1 学習1回目 — コア予測モデル 1〜2 週間

RGB のみで全体深度を学習。YOLO は使わない。以降の詳細モデルの土台になる。

  • 新規 train_scripts/train_lotus_d_core.ps1
  • train_lotus_d.py（--enable_pre_depth_fusion なし）
  • 初期重み: SD2 Base または公式 Lotus-D（どちらを使うかは判断ポイント参照）
  • 出力: output/core-model/checkpoint-XXXXX/
  • NYUv2 / Hypersim val でコアモデル単体の精度を記録
成功基準: 公式 Lotus-D と同等以上の全体 abs_rel。ここが安定してから学習2へ。
2 方式A — オフライン学習データセット構築 3〜7 日

学習2の前に、全学習データへ推論パイプラインと同じ処理を1回実行し、固定データセットを作る。

  • 新規 utils/build_detail_train_dataset.py — Step 1〜3 の一括実行
  • Step 1: 全データに YOLO → detections.json + mask PNG
  • Step 2: コアモデルで global + crop 深度 → lstsq 整合 → pre_depth.npy + valid_mask.npy
  • Step 3: YOLO クラスから class_map.npy を生成
  • 新規 utils/object_pre_depth.py — Step 2 の合成ロジック
  • 新規 utils/detail_train_dataset.py — 事前データを読む DataLoader
  • 出力先: data/detail_train/
成功基準: 全学習画像に pre_depth / valid_mask / class_map が揃い、目視で妥当なこと
3 学習2回目 — 詳細予測モデル 1〜2 週間

コアモデルの重みから開始。事前構築済みデータセットを読み込んで学習。YOLO はループ内で実行しない。

  • 新規 train_scripts/train_lotus_d_detail.ps1
  • 初期重み: 学習1の output/core-model/
  • 改修 train_lotus_d.py — --detail_train_data_dir で事前データを読む
  • --enable_pre_depth_fusion + --enable_object_condition
  • 条件ドロップアウト（30%）: 学習時のみ pre-depth をゼロ化（推論時 YOLO なし対策）
  • ROI 重み付き損失 + 勾配損失
成功基準: ROI abs_rel / boundary F1 がコアモデル単体を上回る。全体 abs_rel は ±5% 以内。
4 推論統合と評価 3〜5 日

コア + YOLO + 詳細の3段推論を統合し、5条件で比較評価。

  • 新規 infer_object_refined_depth.py
  • 比較: 公式 / コアのみ / ROI融合 / 詳細（YOLOあり）/ 詳細（YOLOなし）
  • Hypersim 1024×768 + 高解像度 in-the-wild で主評価
5 アブレーション（必要時） 1 週間

効果が不十分な場合の切り分け実験。

9. リスクと対策

学習1（コア）が不安定
公式 Lotus-D 重みをコアとして流用する選択肢を残す
方式A の前処理に時間がかかる
1回だけ実行。増分更新（新画像のみ再生成）に対応
コアモデル更新時に pre-depth を再生成が必要
コアモデルを固定してから方式Aを実行。バージョンをディレクトリ名に含める
YOLO が Hypersim で検出漏れ
オフライン生成結果を目視確認。GT semantic 版も別途用意して上限比較
学習2でコア性能が劣化
ゼロ初期化 + 条件ドロップアウト + コア単体精度の常時監視
NYUv2 では解像度不足で改善しない
Hypersim val + 高解像度 in-the-wild を主評価に
推論が遅い（YOLO + N クロップ × 2モデル）
Top-K 物体。学習時は事前データ読込、推論時のみ YOLO + コアモデルを実行
クロップ深度のスケール不整合
貼り込み前に必ず lstsq 整合（フェーズ1で実証済み）

10. 実装前の判断ポイント

学習1: コアモデルの出発点

方式特徴推奨
A: 公式 Lotus-D をコアとして採用 学習1をスキップでき、すぐ学習2に進める。研究の独自性は学習2に集約。 時間短縮向け
B: SD2 Base からコアを自前学習 データ・設定を完全にコントロール。2段階が明確になる。 論文で学習過程を示す場合

クラス条件の形式（学習2）

方式特徴推奨
A: class map 画素ごとのクラス ID マップ。空間情報あり。 まずこちら
B: object stats bbox + class + depth のベクトル。シンプルだが空間情報は間接的。 効果が薄い場合に追加

学習2のデータ準備方針

方式特徴採用
方式A: オフライン固定データセット 全データに YOLO + コアモデル推論を1回実行し pre-depth / class_map を保存。推論と同一条件。 採用（本計画）
方式B: 学習時擬似合成 GT 深度を劣化させて pre-depth をその場生成。高速だが推論条件とずれる。 不採用

学習2の YOLO マスクソース（方式A 内）

方式特徴推奨
A: YOLO 事前生成 実写向け。方式A の Step 1 で全データに実行して固定化。 本番用
B: Hypersim GT semantic オラクル品質。方式A の別バージョンとして上限確認用に生成。 上限比較用

評価の主戦場

NYUv2（640×480）は参考程度。 主指標は Hypersim val（1024×768） と 高解像度 in-the-wild 画像 とする。
